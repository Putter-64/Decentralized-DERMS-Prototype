Title of Project: Prototyping and Evaluating for a Decentralized Distributed Energy Resource Management System

 

Abstract: 
The widespread adoption of Distributed Energy Resources (DERs) is critical for grid resilience and energy efficiency, but their expanding digital footprint introduces significant cybersecurity concerns. Traditional DER Management Systems (DERMS) rely on centralized, trust-based models that are vulnerable to single points of failure from individual DERs, and data control issues. This project proposes a paradigm shift by leveraging the Solid protocol to return data ownership and access control to users in the form of a web portal. Said web portal serves as a prototype DERMS interface, enabling secure telemetry monitoring and command transmission for utilities with user-owned Solid Pods as an intermediary data store. The portal authenticates users via WebID and parses SCADA protocols such as Modbus and DNP3 from both simulated and physical DERs including those on Typhoon-HIL, Wokwi, and Raspberry Pi based devices. It also provides several features of COTS DERMS providers such as a graphical user interface for data visualization and control commands sent to devices. By successfully prototyping core DERMS functionalities such as telemetry reading and data graphing using this decentralized framework, this work showcases the feasibility of a user-centric, secure, and interoperable alternative to conventional DERMS architectures.

1 Introduction 
1.1 Motivation
	Recent years have witnessed an unprecedented increase in Distributed Energy Resources (DERs), including rooftop solar panels, battery storage systems, and electric vehicles. These technologies are fundamentally transforming the electrical grid from a centralized, unidirectional system into a decentralized, bidirectional network [1]. This transformation offers substantial benefits to consumers and grid operators alike, such as load balancing capabilities, peak shaving through strategic energy storage and discharge, reduced transmission losses, and accelerated pathways to decarbonization [2].
However, this proliferation of intelligent, connected energy devices also introduces a new set of challenges, particularly regarding security and privacy. Each DER represents a potential entry point into the broader energy grid. If a malicious actor gains access to the grid through a compromised DER, they potentially gain access to telemetry data from other DERs in the system. This data can be analyzed to extrapolate sensitive information such as when residents are home, their daily schedules, which appliances draw significant power, and overall consumption patterns. As DER deployments scale exponentially, the risk posed by insecure communication grows significantly as well [4].
Concurrently with growing security awareness, interest in decentralized data management has intensified across multiple domains. Solid (Social Linked Data), which is an open-source web framework developed by Sir Tim Berners-Lee at MIT, proposes a fundamental shift from traditional centralized data models to one where users maintain ownership and control over their data through personal data storage "pods" [3]. This concept aligns remarkably well with the privacy and security requirements emerging within the DER domain.
The motivation behind this thesis lies in evaluating whether user-owned Solid Pods can support the operational requirements of a centralized DERMS while reducing the privacy and cybersecurity risks associated with them. Rather than replacing field devices outright, this project connects existing industrial Modbus and DNP3 telemetry streams to decentralized native storage and then evaluates whether a utility-facing dashboard could still perform monitoring and control workflows effectively. As DER adoption accelerates, this decentralized architecture offers a practical path to scale visibility and coordination without requiring customers to surrender full ownership of their energy data.
1.2 Problem Statement
Current DERMS and industrial control protocols like Modbus and DNP3 operate on centralized, trust-implicit models. These architectures concentrate vast amounts of sensitive data in single repositories managed by utilities or third-party aggregators [4]. This centralization creates multiple vulnerabilities. One of which is a single point of failure. A successful attack on a centralized DERMS can compromise thousands of endpoints simultaneously [4]. There are also data privacy concerns. Users cede control of their energy consumption data, which can reveal intimate details of their daily lives. There also exists inherent protocol insecurity due to legacy SCADA protocols like Modbus lacking fundamental security features including encryption and authentication [5]. Lastly, there are large scalability limitations as centralized architectures face increasing complexity and cost as DER deployments scale.
Because of these factors, a fundamental redesign is necessary, moving from a centralized trust model to one of decentralized verification and user data sovereignty. The Solid protocol, with its core principles of Personal Online Datastores (Pods) and access control via WebIDs, presents a potentially viable architectural alternative. In this model, each DER owner maintains control over their data in their own Pod, granting more granular access to utilities or aggregators as needed, thereby mitigating the usual risks associated with a centralized data repository.
This project seeks to simultaneously increase DER user data sovereignty while enabling utility visibility and control through a decentralized architecture. By utilizing software to simulate multiple DER devices writing to corresponding Solid Pods and developing a dashboard that aggregates and visualizes this distributed data, this work demonstrates that personal data stores can serve as the foundation for a new class of decentralized DERMS.
1.3 Objectives
The primary objective of this project is to develop and validate a prototype Decentralized DERMS (DDERMS) that leverages Solid personal data stores. In the implemented prototype, the web application authenticates a Solid session using WebID/OIDC, reads telemetry from Pods found on the server, visualizes aggregated Modbus and DNP3 data in graphical format, and provides a basic command-writing path for device control experiments. This prototype is intentionally scoped as a proof-of-concept dashboard rather than a full commercial DERMS replacement, and is used to evaluate whether personal data stores can support decentralized telemetry access and operator interaction.
2 Background
2.1 Distributed Energy Resources (DERs)
Distributed Energy Resources encompass a diverse range of small-to-medium-scale energy technologies located near the point of consumption rather than centralized generation facilities. These include photovoltaic solar panels, wind turbines, battery energy storage systems, electric vehicles with vehicle-to-grid capabilities, controllable loads, and combined heat and power systems [].
DERs fundamentally alter power system dynamics by introducing bidirectional power flows, variable generation patterns, and distributed control challenges. The International Energy Agency projected that DER capacity will have doubled by 2025, representing over 30% of total generation capacity in many regions []. This proliferation necessitates sophisticated management systems to maintain grid stability while maximizing DER value.
From a data perspective, each DER generates continuous streams of telemetry, including power output, state of charge, voltage, frequency, temperature, and operational status. This data is valuable for both grid operations like load forecasting, voltage regulation, contingency planning, as well as end users with consumption awareness and system optimization. However, this same data can also reveal intimate details when aggregated and analyzed.
At the control-interface level, DER inverters are commonly categorized as grid-following or grid-forming. Grid-following devices synchronize to an external voltage and frequency reference and inject power relative to that reference, while grid-forming devices can establish local voltage and frequency setpoints and support islanded operation. In typical utility interconnections, the bulk power system operates at standard AC frequencies like 60 Hz, for example, while many DER subsystems internally use DC buses and converter switching domains at much higher effective frequencies. For the remainder of this paper, DER devices are assumed to generally be grid following.
2.2	 Distributed Energy Resource Management Systems (DERMS)
Distributed Energy Resource Management Systems (DERMS) are platforms designed to monitor, control, and optimize the performance of distributed energy assets []. Traditional DERMS perform several critical functions, such as aggregating and visualizing telemetry from thousands of endpoints to provide grid operators with situational awareness. They also control and regulate DER devices by issuing commands to them for voltage regulation, frequency response, and peak load management. DERMS also allow for optimization, as they coordinate DER operations to maximize economic value while respecting grid constraints, while also predicting DER output and grid conditions for planning and market participation.
Commercial DERMS offerings from vendors including Siemens, Schneider Electric, and AutoGrid typically employ centralized architectures. Data flows from DERs through communication gateways to central servers, where it is stored in databases, processed, and presented to operators. While this is effective, this model creates a weak point for attackers and raises significant privacy concerns.
2.3 Centralized vs. Decentralized Architectures 
Centralized architectures offer simplicity of management and unified data views. However, they introduce fundamental vulnerabilities. A single successful breach can expose data from thousands of customers and potentially enable widespread grid disruption. The 2024 FrostyGoop malware incident, which disrupted heating systems across 600 buildings in Ukraine by exploiting Modbus vulnerabilities, demonstrates the real-world consequences of insecure centralized architectures [6]. This incident highlights how legacy protocols can be exploited in critical infrastructure environments when deployed without adequate protection.
Current ADMS/DERMS deployments often improve resilience at the grid operations layer, yet still rely on centralized data lakes and trust-heavy gateway tiers that aggregate customer telemetry into utility-controlled stores. This architecture can be robust for analytics and dispatch, but it creates persistent governance and exposure challenges: broad internal access surfaces, difficult customer-level data minimization, and high-impact breach domains. In contrast, the architecture in this thesis keeps telemetry resident in customer-associated Pods and moves utilities toward scoped, revocable data access at the resource level. The design therefore preserves operational visibility while reducing the consequences of any single repository compromise. [That one source I had]
Decentralized architectures can distribute data and control across multiple nodes, eliminating single points of failure. Each user maintains authority over their data, granting access only to authorized entities. This aligns with emerging privacy regulations including the General Data Protection Regulation (GDPR) and the California Consumer Privacy Act (CCPA), which grant individuals rights over their personal data. In the energy domain, decentralization has been explored primarily through blockchain-based transactive energy systems [7]. These approaches focus on peer-to-peer energy trading rather than grid management and visibility. This thesis extends decentralization concepts to DERMS functionality itself through Pod-mediated telemetry access and command workflows.
2.4 Protocol Gateways for Legacy DER Integration
Most deployed DER assets are not natively capable of Solid/OIDC interactions, and replacing those devices is typically cost-prohibitive. A practical decentralized DERMS therefore requires protocol gateways that preserve compatibility with field protocols while adding modern identity and access controls at the data-sharing boundary. In this project, Modbus and DNP3 data paths are bridged to Pod resources through software gateways that handle protocol parsing, metadata normalization, and authenticated HTTP operations.
This gateway-first strategy enables phased migration. Existing devices continue operating with familiar protocol semantics, while utilities and customers incrementally adopt Pod-based sharing and access policies. The result is an implementation path that is technically feasible for mixed fleets and organizationally realistic for utilities that cannot perform wholesale hardware refreshes.
2.5 The Solid Protocol and Personal Data Stores
Social Linked Data, or Solid, is an open-source initiative led by Sir Tim Berners-Lee that proposes a fundamental shift from traditional centralized data models to one where users retain ownership and control of their data [9]. Solid enables users to host their data in decentralized Personal Online Data stores, or Pods. Pods are secure data storage containers where individuals and organizations may store their data. Each Pod is identified by a unique URL and controlled by its owner.
WebIDs are identity documents that provide authentication and serve as the foundation for access control. They uniquely identify a user or entity. Access Control Lists provide fine-grained permissions specifying which agents can read, write, or append to specific resources. Linked Data is structured semantically using the Resource Description Framework (RDF), enabling interoperability across different applications and domains. Data is represented as triples: subject, predicate, object, serialized using formats such as Turtle, making data inherently interoperable.
Solid uses Resource Description Framework to represent data, allowing different applications to read and write data using a shared schema without needing a middleman translating information between applications. The authentication flow is similar to OpenID Connect, with a WebID attached before a request is sent to the identity provider that the user prefers, enabling decentralization by assigning a unique identifier to a globally accessible URI.
Solid Pods can store any type of data, but the semantic structure enables automated reasoning and discovery. Applications request access to specific data, and users grant or deny permission through their Pod's access control mechanisms. The Solid ecosystem includes server implementations such as Community Solid Server and Inrupt Enterprise Solid Server, client libraries, and a growing community of applications primarily focused on social media and personal data management [Same solid source prob]. This project seeks to extend Solid's application domain to industrial control systems and critical infrastructure.
3 System Design and Implementation
3.1 Prior Works
	3.1.1 Pranav Natarajan: Solid Client on Constrained Embedded Hardware
Pranav Natarajan's Master's thesis, "Solid Embedded Telemetry," addressed a fundamental question for decentralized DERMS: can a resource-constrained embedded device, the type commonly found in DER deployments, function as a native Solid client capable of both uploading telemetry and responding to remote commands [11]. This question was critical because any practical decentralized DERMS must support devices with limited computational resources, memory, and power budgets.
The thesis demonstrated that an ESP32 microcontroller is capable of executing a full Solid-OIDC v0.2.0 authentication flow and performing authenticated HTTP operations to a Solid Pod server without middleware. The ESP32 was selected as the hardware platform due to its low cost, integrated Wi-Fi connectivity, and widespread community support. The ESP-IDF framework provided cryptographic primitives through mbedTLS, including Elliptic Curve Digital Signature Algorithm operations essential for Solid authentication.
The relevance of Pranav's work to the current project is foundational, since it established that DER-scale embedded devices can function as native Solid clients, performing authentication, telemetry upload, and command reception without requiring intermediary servers or protocol translation layers. This demonstration removed a key uncertainty in the feasibility of decentralized DERMS: if the devices themselves can interact directly with Solid Pods, then the architecture can truly be decentralized, with each device operating as an autonomous agent rather than a subordinate component in a centralized hierarchy.
However, Pranav's work focused exclusively on the device side of the architecture. The ESP32 clients uploaded telemetry and received commands, but ultimately the system lacked a utility-facing dashboard that could aggregate data from multiple devices and present a unified grid view. The current project builds directly on this foundation by developing exactly such a dashboard, extending the decentralized architecture from the device level to the utility operations level.
3.1.2 Donna Thakadipuram: Modbus-to-Solid Protocol Gateway
Donna Thakadipuram's undergraduate honors thesis, "Securing Distributed Energy Resources: A Secure Gateway for Modbus to Solid Communication Using a Raspberry Pi," addressed the critical challenge of integrating legacy Modbus-based DERs into the Solid ecosystem [12]. While Pranav's work demonstrated that new devices could be built as native Solid clients, the reality of DER deployments is that vast numbers of existing devices communicate using Modbus RTU, a protocol that lacks native support for encryption, authentication, or access control. Replacing these devices is economically impractical, so any viable decentralized DERMS must accommodate them.
The thesis developed a communication gateway that enables Modbus RTU-based legacy devices to interact with Solid Pods without modifying the underlying devices. The system centered around a Raspberry Pi 4 configured as a Modbus master, connected to peripheral devices through an RS-485 serial interface. The Raspberry Pi was selected for its affordability, accessibility, and support for Python programming, which enabled rapid prototyping.
The gateway performed two primary functions. For telemetry upload, it polled Modbus peripheral registers at configurable intervals, parsed the raw responses to extract peripheral address, function code, and register data, and then augmented this information with additional context to create semantic resources suitable for publication to a Solid Pod. This contextualization step was critical because raw Modbus register values are opaque numbers that convey meaning only when interpreted against a device-specific register map. The gateway added this meaning, identifying the hardware component associated with each register, interpreting the function code to describe the action performed, converting the register value into a meaningful representation, and including relevant information about the type or unit of the value.
For command reception, the gateway subscribed to a designated command resource in the Solid Pod using WebSocket notifications. When a command was detected, the gateway translated the instruction into a Modbus write operation, constructed the appropriate frame including function code, register address, value, and CRC, and issued it to the peripheral device. A confirmation was subsequently uploaded back to the Pod, completing the communication loop.
The relevance of Donna's work to the current project is equally foundational. It demonstrated that legacy devices, which constitute a significant portion of existing DER deployments, can be integrated into the Solid ecosystem through a gateway architecture. This integration does not require modifying the devices themselves, preserving backward compatibility and economic viability. While Modbus was used during this stage of the project, it has since been extended by Ethan Coffman to support DNP3, as well. The gateway approach bridges the gap between legacy industrial protocols and modern decentralized web technologies, enabling DERs to participate in decentralized DERMS without requiring hardware upgrades, and to allow for simulated DER data, which is critical for the development of the dashboard.
However, Donna's work focused on a single gateway device communicating with a single Solid Pod. The current project extends this by broadening protocol handling (including DNP3-formatted telemetry paths) and by developing a utility-facing dashboard that can be pointed at different Pods by changing the configured Pod name during operation. The protocol gateway concepts from Donna's work inform how individual DERs connect to their Pods, while the dashboard developed in this project demonstrates how a utility-style client can interact with Pod-hosted data.
3.1.3 Prior Work Synthesis and Extension
Together, these prior works establish that Solid-based communication is feasible with both constrained microcontrollers and legacy industrial protocols. Pranav's work demonstrated that new DERs can be built as native Solid clients, operating autonomously without middleware. Donna's work demonstrated that existing Modbus-based DERs can be integrated through gateway architectures, preserving legacy investments while adding modern security capabilities.
The current project synthesizes and extends these foundations. It draws on Pranav's insights about device-side Solid client implementation, including the use of unique resource identifiers to avoid overwrite conflicts and the characterization of latency thresholds for telemetry upload. It draws on Donna's insights about Modbus-to-Solid gateway design, including the semantic enrichment of register values and the use of WebSocket notifications for command delivery.
The novel contribution of this project is the utility-facing DDERMS dashboard implementation that operationalizes Pod-based telemetry access for both Modbus and DNP3 data formats. While the prior works focused on the device-to-Pod communication path, this project adds the operator interface layer via adding Solid login, authenticated data retrieval, parser-based normalization, live chart/table rendering, and multi-Pod aggregation in a single session. The design demonstrates the core decentralized principle that utility software can consume user-owned Pod data without hosting that data centrally.

The proposed decentralized DERMS architecture consists of three primary components: DER devices that generate telemetry data, Solid Pods that store this data under user control, and a DDERMS dashboard that reads and visualizes Pod-hosted telemetry. This design inverts the traditional DERMS model. Rather than data flowing from DERs to a utility-owned central database, data flows from DERs to user-owned Pods, and the utility-style dashboard requests access to read data as needed.
This model has several important implications. First, data sovereignty is preserved: each DER owner maintains control over their data, granting access only to authorized entities. Second, the utility no longer bears the security burden of storing vast quantities of sensitive customer data; that burden is distributed across the Pods themselves. Third, the architecture scales naturally with the number of DERs, as each new DER adds its own Pod rather than contributing to a growing central database. Fourth, the architecture accommodates heterogeneous DER types, from native Solid clients to legacy Modbus devices through gateways, as long as each can publish to its Pod.
The system architecture comprises four layers. The DER device layer includes physical Modbus devices connected through a gateway, native Solid clients such as ESP32 devices, and simulated devices for scalability testing. The protocol bridge layer translates between device-specific protocols and Solid HTTP operations, handling authentication, telemetry upload, and command reception. The Solid Pod layer provides secure, access-controlled storage for semantic telemetry data, with each Pod owned and controlled by the DER owner. The DDERMS dashboard layer provides a browser-based reader interface as the default route, enabling authenticated access, visualization, and limited experimental control interactions against Pod resources.

3.3 DER Integration and Data Acquisition
This project interfaces with two classes of DERs to validate the decentralized architecture across different deployment scenarios. One is a physical Modbus-based device, and the others are simulated environments. The integration of both classes demonstrates that the DDERMS architecture can accommodate heterogeneous device types without requiring uniformity in device implementation.
3.3.1 Physical Modbus Device
 
Figure 4. Custom PCB Modbus peripheral representing a DER device

The physical DER testbed uses a custom printed circuit board as the Modbus peripheral device. This board contains a Texas Instruments TMDSCNCD28335 digital signal processor daughtercard that handles Modbus communications, register operations, and GPIO control. The device is equipped with adjustable potentiometers that emulate input registers, allowing the reading of values corresponding to physical sensor states, and indicator LEDs that represent output registers, providing visual confirmation of values being correctly written.
The gateway is built on a Raspberry Pi 4 running Raspbian OS version 10, with all programming done in Python 3.12.6. Serial communications are managed using the pySerial library. The gateway is configured as a Modbus master and connected to the peripheral device through an RS-485 serial interface.
Telemetry acquisition follows a polling model. The gateway sends Modbus RTU requests to the peripheral device at 500-millisecond intervals, using function code 0x03 to read multiple holding registers. When a response is received through the serial connection, it is passed to a response processing function that extracts the relevant data bytes and converts raw register values to integers using built-in Python data functions.
A key feature of the implementation is a bucketing system that categorizes potentiometer values into ranges by dividing the raw value by 168. This reduces the volume of updates sent to the Solid Pod by only triggering a write operation when the bucket value changes. This logic is implemented using an attribute within the response processing function. If a change is detected, the new telemetry value is transmitted along with context. This method provides a simple form of change detection that ensures minor fluctuations in analog input do not result in excessive activity, which is especially useful in systems with many inputs to prevent overwhelming the Solid Pod.
For command execution, the gateway uses function code 0x06 to write to a single holding register. The request packet is constructed using the function code, peripheral address, register address, and value to be written. A cyclic redundancy check is calculated and appended to the packet, and the request is sent through the serial connection. This write capability is invoked when the Solid Pod sends updated control values to the gateway, enabling bidirectional communication where sensor data can be read from the device and published to the Solid Pod while control commands can be issued remotely.
3.3.2 Simulated DER Environments
3.3.2.1 Wokwi-based ESP32 Simulations
To enable scalability testing without requiring physical deployment of hundreds of devices, the project incorporates simulated DER environments. These simulations serve two main purposes. They allow for the evaluation of system behavior under load with up to thirty-two concurrent devices, and they can enable the replay of extensible telemetry data as mock DER data to test the dashboard's aggregation and visualization capabilities.
The simulation platform uses the wokwi-cli command-line interface, which can allow for scripted and automated simulations of multiple ESP32 instances running concurrently. Each instance is configured with unique environment variables and authentication credentials corresponding to a distinct Solid account. The wokwi-cli tool operates on a session-based model where a simulation duration and configuration file are specified, and the CLI automatically compiles and executes the firmware.
Each simulated ESP32 instance runs the Solid client firmware developed in prior work, which performs authentication, telemetry upload, and WebSocket-based command listening. The simulated devices can be configured to upload telemetry at specified intervals with specified payload sizes, enabling systematic evaluation of how system performance scales with device count.

3.3.3 Semantic Enrichment of Telemetry Data
A critical function performed by both the physical gateway and the simulated devices is the semantic enrichment of raw telemetry data. Raw Modbus register values are opaque numbers that convey meaning only when interpreted against a device-specific register map. The semantic enrichment process adds this meaning, converting raw values into structured resources that can be understood by any application that knows the ontology.
The enrichment process begins with parsing the Modbus response to extract the peripheral address, function code, and register data. The peripheral address uniquely identifies the connected device. The function code defines the requested operation, such as reading or writing registers. The register data holds the value being read or written.
This raw information is then augmented with additional context. The hardware component associated with the register is identified, such as a potentiometer or an LED. The function code is interpreted to describe the action performed, converting numeric codes like 0x03 into human-readable descriptions like "read holding registers." The register value is converted into a meaningful representation, preserving the numeric value while adding context about what it represents. Relevant information about the type or unit of the value is included, such as voltage, current, or state.
The enriched data is then serialized into RDF/Turtle format. A typical resource might include the device URI, timestamp of access, function code description, register function, register number, and value. This context-rich structure allows any authorized application to understand the meaning of the data without requiring prior knowledge of the specific device implementation. For example, a third-party observer reading the resource can immediately understand that a particular register value represents an LED controller and that writing a value of 5 to that register turns on the LED, without needing to consult separate documentation.
The context identifiers are predefined by specification sheets for the Modbus peripheral devices. In a production deployment, device manufacturers would supply the ontologies for producing these TTL resources from streaming telemetry. This approach creates a clear division of responsibilities: device manufacturers define the meaning of their registers, device owners control access to the data, and applications consume the semantically enriched resources without needing device-specific knowledge.
3.4 Solid Pod Configuration
The Solid Pod is responsible for securely storing telemetry data in a decentralized and user-controlled environment. For this project, the Community Solid Server version 6.1.0 is hosted on an AWS EC2 t2.micro instance. The server is configured with WebSocket-based notifications to enable real-time command delivery and with in-memory storage to simplify testing. While in-memory storage may not be suitable for production deployments, it is adequate for prototype validation and avoids the complexity of persistent storage configuration.
Each DER device maintains its own Pod, with the device owner or utility company controlling access permissions, depending on the implementation. For the physical Modbus device, the Raspberry Pi gateway authenticates to the Pod using client credentials and uploads telemetry to a designated resource container. For simulated ESP32 devices, each instance authenticates using its own credentials written in SPIFFS and uploads to its own Pod.
The Solid server implements the Solid-OIDC authentication protocol. On startup, the gateway initializes a CSSAccount object with parameters defined in configuration files, including the Solid server base URL, identity provider endpoint, and user login credentials. The credentials are used to obtain a short-lived access token via get_client_credentials. The resulting token is stored temporarily and reused for all HTTP interactions until expiration.
Access control is implemented through WebID-based permissions. The Pod owner can grant read access to the utility's WebID for specific resources, allowing the DDERMS dashboard to fetch telemetry data. The owner can also grant write access to specific resources for command reception, allowing authorized clients to post instructions. Vincent Tunru’s Penny application was used for the purpose of initially assigning read/write permissions and viewing Pod data outside the dashboard during this project. This fine-grained access control enables the granular sharing of data that is crucial for the decentralized DERMS model.

Figure 5. Solid Pod resource model for telemetry and commands (RDF/Turtle resources and ACL gates)
3.5 DDERMS Dashboard Implementation
The DDERMS dashboard serves as the utility-facing interface, demonstrating that decentralized DERMS functionality can be achieved without requiring the utility to own the underlying data. It was written in the Visual Studio Code IDE and built off the Solid-Pod-Writer repository created by Pranav Natarajan. Built using React, React Router, and Inrupt Solid client libraries, the application is configured to open directly into a reader interface. The reader focuses on telemetry retrieval and visualization while retaining a lightweight in-reader control mechanism for prototype command writes. These command writes represent shutting off devices that display anomalous behavior.
 
Figure 6. Dashboard program architecture (React modules: auth, Pod discovery, parsing/normalization, visualization, command UI)
3.5.1 Authentication and Authorization
Authentication is implemented through WebID-based login using the Inrupt solid-client-authn-browser library. The utility operator logs in via the utility's WebID, obtaining a session that provides authenticated fetch capabilities to Pods. The login flow follows the Solid-OIDC protocol, with the utility operator redirected to their chosen identity provider, which could be a corporate identity provider rather than a consumer-oriented service.
Once authenticated, the dashboard can access Pod resources for which the logged-in WebID has permission. In the implemented interface, the operator specifies one or more Pod names (comma-separated), and each value is used to build target resource paths under the configured Solid server base URI. There is no persistent multi-customer registry UI in the current codebase, so Pod targeting is session-driven and operator-entered rather than stored as a managed customer list.

Write paths are likewise ACL constrained. The command-writing controls are only effective when the active session has explicit write or append access to the target command resource, preventing unauthorized command publication from unauthenticated or underprivileged sessions.
Figure 7. Authentication and authorization flow (WebID/OIDC -> Solid session -> ACL-enforced resource access)
3.5.2 Telemetry Visualization
Telemetry visualization is implemented using authenticated HTTP fetching plus Solid notification WebSockets. The reader discovers telemetry resources under each selected Pod (including recursive discovery under dnp3/devices paths and reads of Modbus resources), fetches those resources, and auto-detects the parser type of Modbus or DNP3. Parsed points are normalized into a common internal data model, tagged by source Pod, and rendered.
For updates, the current implementation opens Solid WebSocket notification channels for each selected Pod, with topics targeting each Pod's Modbus resource. On notification, the reader re-fetches available data and refreshes the displayed charts and tables. This provides near-real-time refresh behavior for active resources while keeping the implementation lightweight.
The visualization layer provides time-series charts via Recharts, per-source/per-device grouping controls, DNP3 family filtering, summary counts, and a raw data table for debugging. 
3.5.3 Command Execution
Command capabilities in the current prototype are implemented as direct file writes to the Pod's Modbus resource path. In the present UI configuration, this is exercised through in-reader device-toggle controls that append plain-text command entries (for example LED_ON or LED_OFF) using authenticated overwrite operations after reading current content. This approach provides a practical end-to-end control path for prototype testing with existing bridges, while also highlighting future work needs around stronger command schemas, acknowledgement records, and idempotent retries.

3.6 Testing Platform
To validate the DDERMS architecture across different deployment scales, the testing platform incorporated both physical hardware and simulated environments. 
The Solid server was hosted on an AWS EC2 t2.micro instance running Community Solid Server version 6.1.0 with WebSocket-based notifications and in-memory storage. The Raspberry Pi 4 bridge ran Raspbian OS version 10 with Python 3.12.6. Simulated ESP32 instances ran in the Wokwi simulation environment using ESP-IDF version 5.2.

Figure 8. End-to-end testing platform overview (physical testbed, Wokwi scalability testbed, Typhoon-HIL microgrid simulation)
3.6.1 Physical Testbed
The physical testbed consists of the custom PCB Modbus peripheral device, the Raspberry Pi 4 gateway, and a local network connection to the AWS-hosted Solid server. This testbed validates end-to-end communication from physical hardware through the gateway to Solid Pods and back.
The testbed is used to measure latency for each stage of the communication flow. Using Python scripts with timing instrumentation, the system records the time from command issuance to Solid Pod write, from Pod write to command reception at the gateway, from command execution to telemetry upload, and from telemetry upload to confirmation reception. These measurements are repeated one hundred times for each configuration to obtain statistically representative results.
3.6.2 Simulated Scalability Testbed
The simulated scalability testbed uses wokwi-cli to run multiple ESP32 instances concurrently. The testbed is configured on a development workstation running Windows 11 with Docker containers orchestrating the simulations. Each ESP32 instance runs the Solid client firmware compiled for the Wokwi environment, with unique credentials and Pod URLs configured through environment variables.
The testbed supports configurations from one to thirty-two concurrent devices. For each configuration, the system measures task execution times for authentication, telemetry upload, WebSocket subscription, and resource fetching. These measurements are logged through the ESP-IDF monitor and aggregated for analysis.
The simulated testbed also enables evaluation of resource contention. By configuring multiple devices to write to the same Pod resource, the testbed reproduces the overwrite conditions identified in prior work and validates the UUID-based resource creation strategy for resolving them.
3.6.3 Microgrid Simulation Testbed
The microgrid simulation testbed uses a Typhoon-HIL system to run high-fidelity simulations of DER-equipped microgrids. The simulation generates realistic telemetry data, including power output variations, voltage fluctuations, and frequency responses. This telemetry is streamed to Solid Pods through a gateway that translates the simulation outputs into RDF/Turtle format.
This testbed validates the DDERMS architecture under realistic grid conditions. The telemetry patterns generated by the simulation reflect actual operational scenarios, including normal operation, ramp events, and fault conditions. The dashboard's ability to aggregate and visualize this realistic data demonstrates that the decentralized architecture can support the same operational use cases as traditional DERMS.
3.7 Data Flow and Command Loop
The complete data flow in the DDERMS architecture follows a consistent pattern across both physical and simulated deployments. Understanding this flow is essential for analyzing system performance and identifying optimization opportunities.



Figure 9. Telemetry upload sequence (DER device/gateway -> authenticated HTTP POST -> Solid Pod -> resource creation/append)
Telemetry flow begins at the DER device. For physical Modbus devices, the Raspberry Pi gateway sends a read request to the peripheral, receives the register values, parses the response, adds semantic context, and constructs an RDF/Turtle resource. For simulated devices, the ESP32 firmware reads simulated sensor values, constructs the resource directly, and initiates the upload.
The gateway or device then performs an authenticated HTTP POST to the appropriate container in the user's Solid Pod. The request includes the access token obtained through Solid-OIDC authentication. The Solid server verifies the token, checks that the authenticated WebID does have write permission for the target container, and stores the resource. If WebSocket notifications are enabled, the server sends a notification to any subscribed clients.

Figure 10. Telemetry read/update sequence (dashboard discover/fetch -> parse -> WebSocket-driven refresh)
The DDERMS dashboard, which has read permission to the selected Pods via stored WebIDs, receives Solid notifications and re-fetches available resources across those Pods. It parses the returned content, extracts telemetry values and metadata, and updates charts, summary statistics, and tabular views.
Figure 11. Command loop sequence (dashboard writes to Pod -> bridge/device execution -> telemetry confirmation -> dashboard refresh)
Command flow in the implemented UI uses authenticated writes to the Modbus file in the Pod. A bridge/client monitoring that resource can translate these updates into device-level actions. The resulting telemetry updates are then fetched by the reader and shown in the interface.
This bidirectional flow still demonstrates the central decentralized property: utility-facing software can interact with DER data and control channels through user-controlled Pods rather than through direct device network access. However, the present implementation is a prototype control loop and not yet a full production-grade command/acknowledgement layer.
4 Results and Discussion
4.1 Scalability Evaluation
The implemented test workflow indicates that the dashboard and Pod-mediated ingestion pipeline remain operational as the number of simulated devices increases, with performance degradation driven primarily by network and server contention rather than parser instability. Across larger simulated runs, authentication and fetch phases contribute a larger fraction of end-to-end delay than local rendering logic, suggesting that optimization effort should prioritize token/session reuse, Pod resource partitioning, and reduced polling pressure where subscriptions are available.
4.2 Comparison with Traditional DERMS
Compared to traditional DERMS designs, this prototype introduces moderate communication overhead through additional web authentication and resource-level access checks, but it materially improves data sovereignty and containment characteristics. In centralized systems, compromise of a core historian or integration tier can expose broad customer populations. In the proposed architecture, data residency is distributed across user Pods and shared through scoped ACL grants, reducing blast radius and enabling revocation at finer granularity. For utility operators, the primary trade-off is increased integration complexity in exchange for stronger privacy and trust boundaries.
5 Future Work
The DDERMS dashboard can be extended with anomaly-detection capabilities that use aggregated telemetry from multiple Pods to identify unusual operating patterns that may indicate device malfunction or cyberattack. This direction builds directly on the present prototype, which already provides authenticated multi-Pod data access, parser-based normalization of Modbus/DNP3 telemetry, and semantically structured Pod resources that can serve as machine-readable inputs for AI workflows.

A key next step is a neurosymbolic pipeline in which edge devices perform lightweight local inference over short buffered windows of raw packets, publish compact semantic state upstream, and allow higher-level symbolic analysis to run over the shared knowledge graph. Because this thesis established the end-to-end path from DER telemetry to Pod-hosted semantic data and utility-side aggregation, the proposed edge-to-cloud AI split can be introduced incrementally without redesigning the core communication architecture.

In this model, each grid-following DER reports whether it is operating within or outside expected bounds, along with contextual state rather than only raw time-series values. Fleet-level reasoning can then be performed with symbolic methods, such as SPARQL query sets that answer operator-facing questions (for example: which DERs were out of scope over the last 30 days, which assets deviated most from weather-adjusted solar expectations, and how a sudden load event could stress the current DER fleet). This layered approach reduces dependence on monolithic multimodal AI by combining edge intelligence with token- and graph-based orchestration at the system level.

This direction creates three AI research tracks for follow-on work: edge AI on DER devices, AI over semantic data exchange in Pods, and symbolic grid-level interpretation/planning from aggregated structured states. A practical milestone is to implement a demonstration workflow for external stakeholders that shows end-to-end fault simulation, semantic state publication, and query-driven operator insight.

Future iterations should also include operator features that are common in production environments, such as a geographic map view and computed system-level metrics. These may include reserve capacity, aggregate ramp capability, and fleet-level constraint margins. These additions would improve situational awareness and make the prototype more representative of practical utility workflows while preserving the user-controlled data model.


6 Conclusion
This project developed and validated a prototype Decentralized DERMS that leverages Solid personal data stores to enable user control over DER telemetry while maintaining utility visibility and control. The architecture changes the traditional DERMS model, with data flowing from DERs to user-owned Pods, and utilities requesting access to read distributed data as needed.
Evaluation demonstrated that Solid-based communication is feasible with both constrained microcontrollers and legacy industrial protocols. While sending SCADA protocols over HTTP introduces latency, the additional security and privacy benefits justify the trade-off for many DER applications.
The successful demonstration of a utility-facing dashboard that authenticates to Solid, reads and parses Pod-hosted Modbus/DNP3 telemetry across multiple Pods, and supports prototype command writes shows that personal data stores can enable meaningful decentralization of DER data access. While the current implementation omits some advanced DERMS features, it establishes that Solid-based architectures can serve as a practical foundation for a decentralized DERMS trajectory in which devices, users, and utilities interact through a shared data space rather than a centralized utility management system.
